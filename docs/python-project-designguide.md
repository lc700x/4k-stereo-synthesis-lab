在创建和维护 Python 工程项目时，一套清晰、可落地的开发与审查规范，不仅能提升代码质量和可维护性，也能显著降低团队协作成本。下面从**编程规范**和**后期审查规范**两个维度，梳理一份可直接用于实践的指南。

---

## 一、编程规范（创建项目时应遵循的）

编程规范的核心目标是：**让代码看起来像同一个人写的**，并降低认知负担。它覆盖风格、结构、文档、类型、错误处理等多个方面。

### 1. 代码风格（PEP 8 + 团队约定）
以 PEP 8 为基准，并用自动化工具强制落地。

- **缩进与空格**：4 个空格，禁止 Tab。运算符两侧、逗号后加空格，括号内侧不加空格。
- **行长度**：推荐 79（文档/注释）~ 99（代码），团队统一即可。常见配置为 `line-length = 88`（配合 Black）。
- **空行**：类和顶层函数之间两空行；类内方法之间一空行；逻辑段落间可用空行分隔。
- **导入规则**：
  - 顺序：标准库 → 第三方库 → 本地导入，每组之间空一行。
  - 绝对导入优先（`import mypkg.module`），避免 `import *`。
  - 每个导入独占一行，除非单行导入同一模块的多个对象（`from os import path, getcwd`）。
- **括号与换行**：长表达式用括号垂直对齐或悬挂缩进，续行增加一级缩进（4 个空格）。
- **字符串引号**：统一单引号或双引号，一般选用双引号（避免转义单引号），用格式化字符串时优先使用 f-string。

### 2. 命名约定（PEP 8）
- **模块/包**：`lower_with_under.py`
- **类**：`CapWords`（`MyClass`）
- **异常**：`CapWords`，通常以 `Error` 结尾（`ValidationError`）
- **函数/方法/变量**：`lower_with_under`
- **常量**：`CAPS_WITH_UNDER`
- **内部/私有**：前导单下划线 `_internal_func`；避免名称冲突用单下划线尾随 `class_`；类私有（名称改写）用双下划线 `__private`，但尽量少用。
- **布尔值**：使用 `is_` / `has_` 前缀（`is_valid`, `has_items`）。

### 3. 文档与注释（PEP 257 + 风格指南）
- **模块级**：文件头部的 docstring 描述模块用途、示例或重要说明。
- **公共函数/类/方法**：必须有 docstring。推荐 Google 风格或 NumPy 风格（便于工具解析生成文档）。
- **注释**：解释“为什么”而非“做什么”。复杂逻辑、非直观的算法必须注释。
- **类型注解**：所有公共接口均需类型注解（PEP 484）。使用 `typing` 模块，如 `Optional`, `Union`, `List[int]` 等，并借助 `mypy` 检查。

```python
from typing import Optional, List

def find_items(query: str, limit: int = 10) -> Optional[List[str]]:
    """根据查询返回最多 limit 个条目，无结果时返回 None。
    
    Args:
        query: 搜索关键词。
        limit: 最大返回条数，默认 10。
        
    Returns:
        匹配的条目列表，或 None。
    """
    ...
```

### 4. 项目目录结构
推荐以功能模块化，常用模板（可参考 `cookiecutter` 数据科学/通用 Python 项目）：
```
project_root/
├── src/                     # 主要包源码 (或直接用包名 myproject/)
│   ├── __init__.py
│   ├── core/
│   ├── utils/
│   └── ...
├── tests/                   # 测试代码，镜像 src 结构
├── docs/
├── scripts/                 # 辅助脚本
├── pyproject.toml           # 项目元数据、构建、工具配置 (PEP 621)
├── requirements/            # 按环境拆分的依赖文件 (可选)
├── Makefile / tox.ini       # 常用任务入口 (可选)
├── .gitignore
├── .pre-commit-config.yaml  # 预提交钩子
├── README.md
└── LICENSE
```

### 5. 编码实践
- **防御性编程**：尽早验证输入，抛出明确的异常（`ValueError`, `TypeError`）。
- **异常处理**：
  - 捕获具体异常，永远不要用裸露 `except:`。
  - 重新抛出时使用 `raise ... from ...` 保留上下文。
  - 尽量让程序快速失败，便于定位问题。
- **单一职责**：函数短小（理想 < 30 行），只做一件事。
- **不可变性**：倾向于返回新对象，少用可变默认参数，慎用全局状态。
- **上下文管理器**：文件、锁、资源等统一使用 `with`。
- **不要重复造轮子**：优先使用标准库和经过检验的第三方包。
- **日志**：用 `logging` 替代 `print`，合理设置等级。

### 6. 依赖与构建
- **依赖管理**：使用 `pyproject.toml`（PEP 621）或 `setup.cfg` 声明依赖，区分 `install_requires` 和 `dev`/`test` 依赖组。
- **锁定环境**：推荐用 `pip-tools`（生成 `requirements.txt`）或 `poetry`/`PDM`（生成锁文件）。
- **虚拟环境**：每个项目独立虚拟环境（`.venv` 并加入 `.gitignore`）。

### 7. 自动化工具链（直接配置为预提交钩子）
- **格式化**：[`black`](https://github.com/psf/black) 零配置自动格式化。
- **导入排序**：[`isort`](https://pycqa.github.io/isort/) 自动整理导入顺序，与 Black 兼容。
- **代码检查**：[`flake8`](https://flake8.pycqa.org/) 或 [`ruff`](https://github.com/astral-sh/ruff)（速度更快，集成了 flake8 规则集和部分 pylint 检查）用于风格与逻辑错误检查。
- **类型检查**：[`mypy`](https://mypy-lang.org/) 严格模式逐步启用。
- **复杂度检查**：可选的 `mccabe`（内置在 flake8 中）或 `radon`，圈复杂度阈值为 10。
- **安全扫描**：`bandit` 检查常见安全漏洞。

**示例 `.pre-commit-config.yaml`（使用 ruff 和 mypy）：**
```yaml
repos:
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.6.0
    hooks:
      - id: ruff
        args: [--fix]
      - id: ruff-format
  - repo: https://github.com/pre-commit/mirrors-mypy
    rev: v1.11.0
    hooks:
      - id: mypy
        additional_dependencies: [types-requests]
```

## 二、后期审查规范（代码审查与持续检查）

代码审查不仅是发现缺陷，更是知识传递和规范一致的保障。审查规范应包含流程、检查项和自动化门禁。

### 1. 审查流程
- **小步提交，提 PR/MR 早**：单次变更控制在 200-400 行以内，降低审查难度。
- **模板化描述**：PR 描述包含：背景、改动内容、测试情况、截图（若涉及 UI）、关联问题号。
- **指定审查人**：至少一名同行，复杂改动可邀请模块负责人。
- **审查工具**：GitHub/GitLab PR/MR，利用评论、建议修改（suggestion）、状态标记。
- **时间预期**：力争 24 小时内给出反馈，紧急变更可通过同步沟通加速。
- **合并策略**：审查通过 + CI 全绿 + 至少一位审批 → 方可合并。推荐 Squash merge 保持主干整洁。
- **持续跟进**：对审查指出的问题，修改后由同一审查人复核。

### 2. 审查检查清单
审查者应逐项确认，不只看 diff，也要拉取代码在脑海中执行。

#### 2.1 设计与架构
- [ ] 解决方案是否与现有设计一致？是否引入了不必要的复杂度？
- [ ] 公共 API 是否设计合理：接口稳定、职责清晰、易于扩展？
- [ ] 改动是否放在正确的位置？是否存在循环依赖或不合理的耦合？
- [ ] 第三方库引入是否必要？评估了维护成本和许可证兼容性。

#### 2.2 功能正确性
- [ ] 逻辑是否正确？是否覆盖了边界条件（空列表、None、大数、极端输入）？
- [ ] 并发/异步场景是否考虑了线程安全或竞态条件？
- [ ] 是否有数据迁移步骤？回滚方案是否考虑？
- [ ] 错误处理是否恰当（异常捕获、用户提示、重试策略）？

#### 2.3 代码质量
- [ ] 符合团队风格和命名约定（PEP 8 + 自定义）。
- [ ] 函数是否简洁？是否存在过长函数、深层嵌套、重复代码？
- [ ] 变量名是否表意清晰，避免缩写？
- [ ] 注释是否准确有价值？文档字符串是否完整且反映最新实现？
- [ ] 类型注解是否提供且准确？

#### 2.4 测试
- [ ] 是否有对应单元测试/集成测试？新增功能覆盖率满足要求（建议 ≥ 80%）。
- [ ] 测试用例是否覆盖正常路径、异常路径、边界值？
- [ ] 测试是否独立、可重复，不依赖执行顺序？
- [ ] CI 中所有测试是否通过？

#### 2.5 安全与性能
- [ ] 是否存在注入风险（SQL、命令行、路径遍历）？
- [ ] 密钥、令牌是否硬编码？敏感数据是否加密或脱敏？
- [ ] 是否引入了明显性能瓶颈（循环内数据库查询、内存不当保留）？
- [ ] 数据库查询是否使用了索引、避免了 N+1 问题？是否有慢查询风险？

#### 2.6 可维护性
- [ ] 日志是否记录了必要的调试信息和关键变量（避免泄露隐私）？
- [ ] 是否有配置项需要硬编码？是否可通过环境变量或配置中心调整？
- [ ] 向后兼容性是否得到处理（API 弃用周期）？
- [ ] 文档（README、API 文档、运维手册）是否同步更新？

### 3. 自动化检查（CI/CD 门禁）
把 80% 的风格和低级错误拦截在代码审查之前，让审查者聚焦高层次问题。

**CI 流水线必备步骤（示例：GitHub Actions / GitLab CI）**：
1. **格式与导入**：`ruff format --check` 或 `black --check`，`isort --check-only`
2. **静态分析**：`ruff check` 或 `flake8`
3. **类型检查**：`mypy --strict`（或逐步启用）
4. **安全扫描**：`bandit -r src/`
5. **单元测试 + 覆盖率**：`pytest --cov --cov-report=term --cov-fail-under=80`
6. **文档构建（可选）**：`mkdocs build` 或 `sphinx -W` 确保无警告
7. **依赖审计（定期）**：`pip-audit` 或 `safety check` 扫描已知漏洞

只有上述全部通过，PR 才允许合并。可使用 **GitHub Branch Protection** 或 **GitLab Protected Branches** 强制执行。

### 4. 审查文化建议
- **对事不对人**：评论应客观，例如“建议将变量 `x` 改为 `user_count` 以提高可读性”而非“你起名太差了”。
- **区分严重程度**：使用标签如 `nit`（建议性微调）、`blocking`（必须修改）帮助提交者排优先级。
- **鼓励双向讨论**：审查者的理解可能也有偏颇，允许提交者解释设计决策。
- **知识共享**：对新加入成员安排“结对审查”或“代码走读”，加速规范认同。

## 总结与落地步骤
1. **项目初始化**：使用 `cookiecutter` 生成包含上述配置的模板，或手动配置 `pyproject.toml`、`.pre-commit-config.yaml` 和 CI 文件。
2. **团队对齐**：在一份 `CONTRIBUTING.md` 或内部 Wiki 中沉淀本规范，并在站会中花 10 分钟集体学习。
3. **逐步收紧**：对遗留代码设置工具基线（如先只检查新文件），并逐步提升覆盖率门槛。
4. **定期回顾**：每季度检视规范是否需要更新，纳入更好的工具（如迁移到 `ruff` 提升速度）。

遵循这套规范，能让 Python 工程项目从单人作战平稳过渡到团队协作，并在持续迭代中保持代码的长期健康。
