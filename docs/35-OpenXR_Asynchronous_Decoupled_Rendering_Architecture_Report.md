# OpenXR 异步渲染架构设计与实现报告

## 1. 引言与需求分析

在 OpenXR 应用中，我们需同时呈现以下内容：

- 一个复杂的**3D 虚拟房间模型**作为环境背景。
- 一个始终清晰流畅的**虚拟显示器画面**（例如桌面镜像或视频播放），作为房间内的 3D 屏幕平面参与 Projection Layer 渲染。
- 围绕显示器的**光晕特效（Glow）**、文字信息面板、虚拟键盘，以及屏幕内容投射到墙壁上的**动态光反射**。

关键性能目标：**无论房间模型多复杂、光效计算量多大，虚拟显示器的刷新率必须始终稳定在目标帧率（如 90 Hz），不受任何背景渲染负载的拖累。**

本报告将系统性地阐述一种完全异步、零阻塞的渲染架构。由于 Virtual Desktop 对 OpenXR Quad Layer 的交换链格式存在限制，虚拟显示器主画面不再强行放入 Quad Layer，而是回到 Projection Layer 内的 3D 屏幕平面；Quad Layer 专注承载 Glow、文字信息面板、虚拟键盘等轻量 2D 覆盖元素，最终实现“主画面硬实时，软效果全后台化”的目标。

## 2. 核心技术原理：GPU 异步多队列

现代图形 API（Vulkan / D3D12）支持**多个命令队列**并行执行：

- **图形队列**：负责主渲染工作，如绘制虚拟显示器、合成背景。
- **计算队列**：可独立执行通用计算，如降采样、模糊等。
- **拷贝队列**：进行资源传输。

这些队列能够**真正硬件并行**，只要处理好资源依赖。核心设计思想：**将高开销但可容忍延迟的操作卸载到计算队列，主图形队列仅做最少、最紧急的绘制，并通过消费“已完成”的旧结果来避免等待。**

## 3. Glow 特效的完全异步生成

### 3.1 设计思路

Glow 本质是对画面进行大幅模糊后叠加，对单帧延迟极不敏感。我们采用**隔帧消费旧纹理**的方案，使 Glow 成为纯粹的后台进程。

### 3.2 实现流程

```
帧 N 主图形队列：
  1. 渲染虚拟屏幕内容至 tex_screen[N]
  2. 合成阶段使用 tex_glow_safe 叠加光晕（来自帧 N-2 或更早）
  3. 提交命令，不等待任何计算队列信号

帧 N 计算队列（与上一步并发）：
  接收 tex_screen[N] → 降采样到小纹理 → 多次模糊 → 结果写入 tex_glow[next]
  完成时标记该纹理为“safe”，供未来帧使用
```

- **多缓冲池**：维护 3 张 glow 纹理循环使用，确保主队列读取的纹理已确定无写入冲突。
- **无同步开销**：主队列从不等待计算完成信号，仅通过指针轮换指向早已安全的旧纹理。
- **可降频更新**：计算队列甚至可每 2~3 帧才触发一次，节省算力。

该方案被形象地称为“废物利用”——即将已不再需要的旧屏幕纹理交给后台生成光晕，主渲染零成本获取效果。

### 3.3 目标帧时间模型

引入异步处理后的目标形态是：主图形队列只消费已经完成的效果纹理，不在当前帧生成 Glow 或墙面反射。以 90/120 Hz 头显为目标时，主线程预算必须稳定、短小、无等待点。

```text
主图形队列（每帧）：
├── 采样 safe 房间背景纹理 / panorama / equirect          0.5ms
├── 绘制虚拟屏幕 3D 平面 + controller / laser            1.3ms
├── 采样上帧已完成的墙面光斑纹理并合成                  0.7ms
├── 提交 Glow / 文字面板 / 虚拟键盘 Quad 覆盖层          0.5ms
└── 提交 xrEndFrame
总耗时：约 3.0-3.5ms

异步计算队列（与主队列并行）：
├── 接收当前帧屏幕 GPU 纹理
├── 降采样 + 模糊生成新 Glow 纹理                       3.0ms
├── 生成新墙面反射光斑纹理                              1.0ms
└── 原子更新 latest safe effect index
```

关键要求：即使异步计算队列超时，主图形队列也只继续使用旧的 `safe_glow_texture` 与 `safe_light_texture`。Glow 延迟 2~3 帧更新可以接受，主画面帧时间不能因此抖动。

## 4. Projection 主屏与 Quad 覆盖层的职责分离

### 4.1 关键前提

Virtual Desktop 下 Quad Layer 对颜色格式较严格，虚拟显示器主画面直接作为 Quad Layer 时会遇到兼容性风险。因此目标架构采用更稳的分工：**Projection Layer 承载沉浸式 3D 世界和虚拟屏幕主画面，Quad Layer 只承载轻量 2D 覆盖层**。

### 4.2 虚拟显示器回到 Projection 内 3D 平面

虚拟显示器作为 Projection Layer 内的 3D 四边形绘制：

- 3D 房间、全景背景、手柄模型、laser、虚拟屏幕主画面同属 Projection Layer。
- 虚拟屏幕每帧采样最新桌面/立体画面纹理，房间背景可静态或低频更新。
- 屏幕可参与深度测试，必要时被手柄或场景物体正确遮挡。
- 主画面不依赖 Quad Layer 的格式兼容性，避免 Virtual Desktop 对 sRGB Quad swapchain 的限制。

### 4.3 Quad 层承载 Glow 与 2D UI

`XR_KHR_composition_layer_quad` 仍保留，但职责改为轻量覆盖层：

- Glow 光效：使用异步生成的低频/模糊纹理，延迟一两帧也无感。
- 文字信息面板：低频更新，保持清晰，不受 Projection 抗锯齿或房间渲染影响。
- 虚拟键盘：按交互需求独立更新和提交。
- Quad Layer 在 Virtual Desktop 下优先使用兼容性更好的 `GL_RGBA8`；Glow 在线性空间生成并以覆盖/加法风格合成。

### 4.4 Projection 内部异步房间背景渲染

Projection Layer 本身也要拆成硬实时前景与软实时背景两条路径。复杂 3D 房间模型不应每帧直接压在主渲染循环里，而应由异步图形队列渲染到离屏 `room_background_texture` 环形缓冲；主队列每帧只采样最近一次已经安全完成的背景纹理。

**主队列每帧职责：**

```text
xrWaitFrame -> xrBeginFrame
获取 Projection Eye Texture
1. 读取 latestReadyBgIdx，采样已完成的 room background texture
2. 绘制虚拟屏幕 3D 四边形，采样最新桌面/立体纹理
3. 绘制 controller、laser、边框等前景元素
4. 提交 Glow / UI / 键盘 / HDR 2D image 等 Quad 覆盖层
5. xrEndFrame
```

**异步房间背景队列职责：**

```text
1. 从 2~3 张 room background texture 环形缓冲中选择未被主队列读取的 slot
2. 使用当前最新 head pose 渲染完整 3D 房间模型到该离屏纹理
3. GPU 完成后由 CPU 侧 poll fence/signal
4. 原子更新 latestReadyBgIdx，供后续主队列消费
```

关键约束是：主队列不等待背景渲染 fence，也不等待异步队列产出当前帧结果；它只读取已经 safe 的旧纹理。若背景 worker 暂停或超时，主画面继续使用上一张 safe 背景纹理。

触发策略可以按成本选择：每 2~3 帧更新一次、头部旋转超过阈值（例如 0.5 度）时更新，或在 GPU 余量足够时持续提交。首帧必须准备黑色、默认全景或预渲染初始背景，避免读取未定义纹理。

### 4.5 静态 panorama/equirect 是异步背景的低成本特例

当房间完全静态，或已由 Unity、Blender 等外部工具导出 360° 全景纹理时，`room_background_texture` 可以退化为一张长期复用的 panorama/equirect/cubemap 资产：

- 加载阶段读取外部导出的高分辨率全景图。
- 每帧主渲染绘制天空球或全屏背景 pass，根据视线方向采样全景图。
- 若 runtime 支持 `XR_KHR_composition_layer_equirect2` 等扩展，可将全景图直接提交为独立背景层。

因此目标架构不是“只能 panorama”，而是：复杂房间走异步背景纹理环形缓冲；静态房间或 HDR panorama 走 Projection/equirect/sky sphere 的低成本路径。

## 5. 屏幕光墙面反射的动态叠加

### 5.1 问题分析

房间背景资产是静态的，但屏幕内容不断变化，会在墙壁上产生动态的彩色光照。若重新渲染房间将破坏性能优势，因此墙面反射应由随背景资产绑定的掩码/UV 映射与异步生成的低频光源结果共同合成。

### 5.2 预计算 + 动态光斑纹理

**步骤：**

1. **预计算光斑掩码**：离线（或加载期）通过简单光线追踪或几何分析，生成一张与全景图 UV 对齐的静态权重纹理，表示屏幕光对各像素的照射强度（考虑距离衰减、墙面 BRDF 近似）。
2. **异步生成光斑颜色**：在计算队列中，对当前屏幕内容极度降采样（如 1×1 平均色），得到模糊光源颜色。这可以复用 Glow 降采样链，或独立完成，同样允许延迟一帧。
3. **叠加**：在背景绘制的像素着色器中，采样全景纹理得房间色，再用**相同 UV** 采样动态光斑纹理（将平均色与掩码相乘得到），最终以加法混合： `finalColor = 全景色 + 动态光斑色`。

该操作仅增加一次纹理采样和一次加法，对性能无影响，且光斑随屏幕内容柔和变化，视觉上真实可信。

## 6. 综合架构总览

下图展示了最终的异步渲染流水线，所有组件并行不悖：

| 组件 | 刷新策略 | 运行时开销 | 与主显示器同步关系 |
|------|----------|------------|-------------------|
| **房间背景** | 异步房间纹理环形缓冲；静态场景可退化为 panorama/equirect/sky sphere | 主队列只采样 safe 纹理 | 消费旧背景，无等待 |
| **屏幕光墙面反射** | 预计算掩码 + 异步动态光斑 | 纹理采样+加法，可忽略 | 消费旧帧，无等待 |
| **虚拟显示器** | Projection 内 3D 四边形，每帧更新 | 一次纹理采样 + quad draw | 最新帧，零等待 |
| **手柄/controller/laser** | Projection 前景元素，每帧轻量绘制 | 小模型/线段 draw | 硬实时，无等待 |
| **Glow / UI / 虚拟键盘 / HDR 2D 图像** | Quad 覆盖层，消费旧 safe 纹理或低频 UI 纹理 | Runtime 合成，轻量 | 消费旧帧，无等待 |

执行模型（每帧主渲染线程）：

1. 采样 `latestReadyBgIdx` 指向的 safe 房间背景纹理，或采样 panorama/equirect/sky sphere 背景。
2. 在 Projection Layer 内绘制虚拟屏幕 3D 四边形，采样最新桌面/立体画面纹理。
3. 绘制 controller、laser、屏幕边框等硬实时前景元素。
4. 提交 Glow、文字面板、虚拟键盘、HDR 2D 图像等 Quad 覆盖层，使用上一帧或更早的 safe 纹理。
5. 提交 `xrEndFrame`，同时携带 Projection Layer 与可用 Quad 覆盖层。
6. 异步图形队列继续渲染下一张房间背景纹理；异步计算队列处理 Glow 与光斑颜色。两者都不阻塞当前帧。

## 7. 实现注意事项

- **资源同步**：异步房间背景队列、Glow 计算队列与主图形队列之间必须使用信号量/栅栏保护写入完成状态，但主渲染绝不等待这些信号量；始终由 CPU 侧记录的 `latestReadyBgIdx` / safe index 选择已经完成的旧纹理。
- **环形缓冲保护**：房间背景纹理至少 double/triple buffer，异步队列不得写入主队列当前正在读取的 slot。
- **初始帧处理**：启动时房间背景、Glow 纹理与光斑纹理需初始化为黑色、默认全景或透明，防止合成时读取未定义数据。
- **3DoF 前提**：当前目标场景以原地转头为主，没有平移视差；异步房间背景使用稍早 head pose 的延迟通常可接受。未来若支持 6DoF 移动，需要 depth reprojection 或更精细的背景重投影策略。
- **全景图分辨率**：应根据头显单眼分辨率和视场角反推，确保正视方向清晰度不低于 1:1 像素映射。通常 equirect 宽度需达 8K 以上。
- **扩展兼容性**：若目标设备不支持全景背景扩展，背景可退化为天空球方案；若 Quad 层格式受限，虚拟显示器主画面仍由 Projection 3D 平面负责，Glow/UI Quad 可透明禁用并记录原因。
- **光斑掩码制作**：可利用一次性的 GPU 光线投射或借助建模软件离线烘焙，生成全景 UV 空间的单通道纹理。对于简单房间几何，直接在 shader 中通过数学公式计算投影也可行。

## 8. 总结

本架构充分利用 OpenXR Projection Layer、Quad Layer 与现代 GPU 多队列并行特性，将**虚拟显示器主画面**、**异步房间背景**、**动态光效/UI 覆盖层**按职责解耦。核心思想可概括为：

- **严格区分实时路径与后台路径**：显示器更新为硬实时，其余均为软实时并允许延迟。
- **Projection 内部也异步化**：复杂房间模型由后台图形队列渲染到 safe 背景纹理，主队列只负责采样和前景绘制。
- **静态预计算与动态叠代结合**：静态 panorama/equirect 是背景异步管线的低成本特例，动态光效仅以极低成本纹理叠加实现。
- **消费“旧结果”而非等待“新结果”**：房间背景、Glow、墙面反射均采用已完全就绪的前期产物，杜绝任何阻塞可能。

最终效果：虚拟显示器在 Projection Layer 内稳定绘制，Glow 与 UI Quad 覆盖层即使延迟或失败也不阻塞主画面提交，用户获得稳定流畅的 VR 桌面体验。该设计同样适用于任何需要将核心交互元素与高开销软效果分离的 OpenXR 应用场景。
