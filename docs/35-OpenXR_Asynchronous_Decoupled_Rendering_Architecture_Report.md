# OpenXR 异步渲染架构设计与实现报告

## 1. 引言与需求分析

在 OpenXR 应用中，我们需同时呈现以下内容：

- 一个复杂的**3D 虚拟房间模型**作为环境背景。
- 一个始终清晰流畅的**虚拟显示器画面**（例如桌面镜像或视频播放），且**不能被房间内任何物体遮挡**。
- 围绕显示器的**光晕特效（Glow）**，以及屏幕内容投射到墙壁上的**动态光反射**。

关键性能目标：**无论房间模型多复杂、光效计算量多大，虚拟显示器的刷新率必须始终稳定在目标帧率（如 90 Hz），不受任何背景渲染负载的拖累。** 同时用户仅能原地转头（3DoF），无位置移动，这为进一步优化提供了基础。

本报告将系统性地阐述一种完全异步、零阻塞的渲染架构，将上述所有效果与主显示器渲染管线彻底解耦，最终实现“显示器永远流畅，其它效果全后台化”的目标。

## 2. 核心技术原理：GPU 异步多队列

现代图形 API（Vulkan / D3D12）支持**多个命令队列**并行执行：

- **图形队列**：负责主渲染工作，如绘制虚拟显示器、合成背景。
- **计算队列**：可独立执行通用计算，如降采样、模糊等。
- **拷贝队列**：进行资源传输。

这些队列能够**真正硬件并行**，只要处理好资源依赖。核心设计思想：**将高开销但可容忍延迟的操作卸载到计算队列，主图形队列仅做最少、最紧急的绘制，并通过消费“已完成”的旧结果来避免等待。**

## 3. Glow 特效的完全异步生成

### 3.1 设计思路

Glow 本质是对画面进行大幅模糊后叠加，对单帧延迟极不敏感。我们采用**隔帧消费旧纹理**的方案，使 Glow 成为纯粹的后台进程。

### 3.2 实现流程

```
帧 N 主图形队列：
  1. 渲染虚拟屏幕内容至 tex_screen[N]
  2. 合成阶段使用 tex_glow_safe 叠加光晕（来自帧 N-2 或更早）
  3. 提交命令，不等待任何计算队列信号

帧 N 计算队列（与上一步并发）：
  接收 tex_screen[N] → 降采样到小纹理 → 多次模糊 → 结果写入 tex_glow[next]
  完成时标记该纹理为“safe”，供未来帧使用
```

- **多缓冲池**：维护 3 张 glow 纹理循环使用，确保主队列读取的纹理已确定无写入冲突。
- **无同步开销**：主队列从不等待计算完成信号，仅通过指针轮换指向早已安全的旧纹理。
- **可降频更新**：计算队列甚至可每 2~3 帧才触发一次，节省算力。

该方案被形象地称为“废物利用”——即将已不再需要的旧屏幕纹理交给后台生成光晕，主渲染零成本获取效果。

## 4. 虚拟显示器与房间背景的彻底解耦

### 4.1 关键前提

虚拟显示器**永远在顶层，不被任何房间物体遮挡**。这允许我们使用 OpenXR 原生合成层机制，将两者物理分层。

### 4.2 采用 Quad 层分离显示器

使用 `XR_KHR_composition_layer_quad` 将虚拟显示器作为独立图层提交：

- 显示器拥有自己的交换链，每帧更新为最新画面。
- 房间作为投影层（`XR_KHR_composition_layer_projection`）或专用背景层。
- Runtime 合成时，Quad 层始终置于投影层之前，无遮挡问题。
- **帧率完全解耦**：房间层可极低频率渲染（甚至静态），显示器仍满帧率运行，无任何延迟。

### 4.3 房间背景的零渲染方案：预渲染全景纹理

由于用户只能原地转头（无平移视差），观察点完全固定。透视关系仅取决于方向，因此可将整个房间场景**一次性**渲染到一张 360° 全景纹理（如 Equirectangular 或 Cubemap），运行时背景绘制仅为一个极低开销的天空球采样。

- 加载阶段用单独管线渲染房间到高分辨率全景图。
- 每帧主渲染绘制包围球体，像素着色器根据视线方向采样全景图。
- 运算量：单个 draw call，仅纹理采样，对帧预算影响可忽略不计。

若 Runtime 支持 `XR_KHR_composition_layer_equirect2` 等扩展，甚至可将全景图直接提交为独立背景层，连天空球 draw call 都省去，达到真·零渲染背景。

## 5. 屏幕光墙面反射的动态叠加

### 5.1 问题分析

房间是静态的，但屏幕内容不断变化，会在墙壁上产生动态的彩色光照。若重新渲染房间将破坏性能优势。因观察点固定，光斑在墙上的投影位置永恒不变。

### 5.2 预计算 + 动态光斑纹理

**步骤：**

1. **预计算光斑掩码**：离线（或加载期）通过简单光线追踪或几何分析，生成一张与全景图 UV 对齐的静态权重纹理，表示屏幕光对各像素的照射强度（考虑距离衰减、墙面 BRDF 近似）。
2. **异步生成光斑颜色**：在计算队列中，对当前屏幕内容极度降采样（如 1×1 平均色），得到模糊光源颜色。这可以复用 Glow 降采样链，或独立完成，同样允许延迟一帧。
3. **叠加**：在背景绘制的像素着色器中，采样全景纹理得房间色，再用**相同 UV** 采样动态光斑纹理（将平均色与掩码相乘得到），最终以加法混合： `finalColor = 全景色 + 动态光斑色`。

该操作仅增加一次纹理采样和一次加法，对性能无影响，且光斑随屏幕内容柔和变化，视觉上真实可信。

## 6. 综合架构总览

下图展示了最终的异步渲染流水线，所有组件并行不悖：

| 组件 | 刷新策略 | 运行时开销 | 与主显示器同步关系 |
|------|----------|------------|-------------------|
| **房间全景背景** | 静态纹理，一次生成 | 单次天空球 draw call（或零） | 完全独立，无等待 |
| **屏幕光墙面反射** | 预计算掩码 + 异步动态光斑 | 纹理采样+加法，可忽略 | 消费旧帧，无等待 |
| **虚拟显示器** | Quad 层，每帧更新 | Runtime 合成，几乎无开销 | 最新帧，零延迟 |
| **显示器 Glow** | 异步计算，消费旧屏纹 | 仅叠加采样现成纹理 | 消费旧帧，无等待 |

执行模型（每帧主渲染线程）：

1. 绘制天空球（采样全景图+动态光斑纹理）至投影层 Eye Texture。
2. 更新 Quad 层纹理为最新桌面画面（若需要可在此先合成已完成的 Glow 纹理）。
3. 提交 `xrEndFrame`，同时携带投影层与 Quad 层。
4. 触发计算队列任务（异步处理 Glow 与光斑颜色），这些操作与当前帧提交完全重叠。

## 7. 实现注意事项

- **资源同步**：计算队列与图形队列间必须使用信号量/栅栏保护纹理的写入完成状态，但主渲染绝不等待这些信号量——始终由 CPU 侧记录的“最近安全索引”来选取纹理，保证读取的资源早已处于 `SHADER_READ_ONLY_OPTIMAL` 状态。
- **初始帧处理**：启动时 Glow 纹理与光斑纹理需初始化为黑色或透明，防止合成时读取未定义数据。
- **全景图分辨率**：应根据头显单眼分辨率和视场角反推，确保正视方向清晰度不低于 1:1 像素映射。通常 equirect 宽度需达 8K 以上。
- **扩展兼容性**：若目标设备不支持 Quad 层或全景背景扩展，可退化：显示器绘制为投影层内四边形，背景仍可用天空球方案，整体架构无需大改。
- **光斑掩码制作**：可利用一次性的 GPU 光线投射或借助建模软件离线烘焙，生成全景 UV 空间的单通道纹理。对于简单房间几何，直接在 shader 中通过数学公式计算投影也可行。

## 8. 总结

本架构充分利用 OpenXR 合成层能力与现代 GPU 多队列并行特性，将**虚拟显示器**与**静态房间背景**、**动态光效**完全异步解耦。核心思想可概括为：

- **严格区分实时路径与后台路径**：显示器更新为硬实时，其余均为软实时并允许延迟。
- **静态预计算与动态叠代结合**：房间背景固化，动态部分仅以极低成本纹理叠加实现。
- **消费“旧结果”而非等待“新结果”**：所有异步效果均采用已完全就绪的前期产物，杜绝任何阻塞可能。

最终效果：即使房间场景极其复杂、光效计算再繁重，虚拟显示器的帧率始终坚挺，延迟为零，用户获得稳定流畅的 VR 桌面体验。该设计同样适用于任何需要将核心交互元素与高开销环境分离的 OpenXR 应用场景。
